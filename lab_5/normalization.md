## 1. Початкова схема (Аналіз існуючих структур)
Початковий дизайн бази даних (з Лабораторної №2) проектувався з використанням сурогатних первинних ключів (`SERIAL`). Це автоматично усуває низку проблем ранніх етапів нормалізації (наприклад, часткові залежності), але логічна структура даних все ще містить приховані аномалії, які порушують правила 1NF та 3NF.

**Проблемні таблиці початкової схеми:**
1. `doctor` ( `doctor_id`, `staff_type`, `specialization`, `people_id` )
2. `service` ( `service_id`, `category`, `title`, `price`, `duration` )

**Виявлені проблеми та аномалії:**
* **Порушення атомарності (1NF):** Атрибут `specialization` має тип `varchar(32)` і на практиці може містити множинні значення (наприклад: "Хірург, Травматолог"). Це унеможливлює коректний пошук, фільтрацію та підрахунок лікарів за фахом.
* **Аномалія оновлення / Транзитивність (3NF):** Атрибут `category` у таблиці послуг залежить від самої послуги (`title`), а не від ключа. Зберігання категорій у вигляді `ENUM` або прямого тексту унеможливлює додавання нової категорії без існуючих послуг (аномалія вставки) і вимагає складних операцій при зміні назви категорії.

## 2. Функціональні залежності (ФЗ)
Згідно з математичною теорією реляційних баз даних, визначаємо повний набір функціональних залежностей для нашої схеми. 
*(Позначення: детермінант -> залежний атрибут)*

**2.1. Залежності сутностей "Людина" та "Пацієнт":**
1. `people_id` -> { `first_name`, `last_name`, `birth_date`, `phone_number`, `email` }
2. `patient_id` -> { `gender`, `type_client`, `people_id` }

**2.2. Залежності сутності "Лікар" (До нормалізації):**
3. `doctor_id` -> { `staff_type`, `specialization`, `people_id` }
*(Проблема: `specialization` приховує в собі множину значень, порушуючи базові принципи ФЗ).*

**2.3. Залежності сутності "Послуга" (До нормалізації):**
4. `service_id` -> { `title`, `price`, `duration`, `category` }
**Транзитивна залежність (залежність між неключовими атрибутами):**
5. `title` -> `category` *(Назва послуги визначає її медичну категорію. Таким чином, маємо ланцюг: `service_id` -> `title` -> `category`).*

**2.4. Залежності "Прийому" (Повна функціональна залежність):**
6. `appointment_id` -> { `date_time`, `patient_id`, `doctor_id` }
7. `appointment_details_id` -> { `appointment_id`, `service_id` }

## 3. Етапи глибокої нормалізації (1NF – 5NF)

### 3.1. Перехід до 1NF (Перша нормальна форма)
**Вимога:** Кожен атрибут повинен містити лише атомарні (неподільні) значення. Жодних масивів, списків чи груп атрибутів, що повторюються.
**Рішення:** Поле `specialization` у таблиці `doctor` є порушенням. Ми декомпозуємо структуру, виносячи спеціальності в окремий довідник і створюючи таблицю для реалізації зв'язку "Багато-до-Багатьох".
* **Нова таблиця 1:** `specialty` ( `specialty_id` PK, `title` )
* **Нова таблиця 2:** `doctor_specialty` ( `doctor_id` FK, `specialty_id` FK )
* З таблиці `doctor` **видаляється** поле `specialization`.

### 3.2. Перехід до 2NF (Друга нормальна форма)
**Вимога:** Схема знаходиться в 1NF, і всі неключові атрибути повністю функціонально залежать від усього первинного ключа (усунення часткових залежностей від складеного ключа).
**Аналіз:** У нашій базі даних використовуються одинарні сурогатні ключі (`SERIAL`). Часткові залежності можливі *лише* за наявності складених первинних ключів. 
*(Теоретичний приклад: Якби таблиця `appointment_details` мала складений ключ `(appointment_id, service_id)` і атрибут `base_price`, що залежить лише від `service_id`, це було б порушенням 2NF).*
**Висновок:** Оскільки в наших основних таблицях ключі одинарні, а в таблиці `doctor_specialty` немає додаткових атрибутів, схема **автоматично відповідає 2NF**.

### 3.3. Перехід до 3NF (Третя нормальна форма)
**Вимога:** Схема знаходиться в 2NF, і відсутні транзитивні залежності неключових атрибутів від первинного ключа (неключовий атрибут не повинен залежати від іншого неключового).
**Рішення:** Усуваємо виявлену транзитивну залежність `service_id` -> `title` -> `category`. Виносимо категорії в незалежний довідник.
* **Нова таблиця:** `service_category_dict` ( `category_id` PK, `name` )
* У таблиці `service` замінюємо текстове поле (чи ENUM) `category` на зовнішній ключ `category_id` (FK). Тепер категорія залежить безпосередньо від свого ідентифікатора.

### 3.4. Перехід до BCNF (Нормальна форма Бойса-Кодда)
**Вимога:** Для кожної нетривіальної функціональної залежності `X -> Y`, детермінант `X` обов'язково є надключем. Це "сувора" 3NF, яка вирішує проблеми перекриття кандидат-ключів.
**Аналіз:** Перевіряємо таблицю `appointment` (`appointment_id`, `date_time`, `patient_id`, `doctor_id`). Чи існують тут приховані детермінанти? Ні, пацієнт не визначає лікаря жорстко, а час не визначає пацієнта. Всі залежності виходять тільки від `appointment_id` (який є суперключем). 
**Висновок:** Схема повністю відповідає BCNF.

### 3.5. Перехід до 4NF (Четверта нормальна форма)
**Вимога:** Схема в BCNF і не містить незалежних багатозначних залежностей (MVD: `X ->> Y`).
**Аналіз:** Багатозначна залежність виникає, коли в одній таблиці намагаються зберегти дві незалежні множини. Якби ми для лікаря зберігали і його спеціальності, і мови, якими він розмовляє, у єдиній таблиці (Лікар, Спеціальність, Мова), ми б отримали декартовий добуток та дублювання даних.
**Висновок:** Завдяки тому, що ми винесли спеціальності в ізольовану таблицю `doctor_specialty` на етапі 1NF, ми уникнули цієї проблеми. Схема відповідає 4NF.

### 3.6. Перехід до 5NF (П'ята / Проекційно-з'єднувальна НФ)
**Вимога:** Схема в 4NF і не може бути декомпозована на менші таблиці без втрати інформації (або без появи "хибних" рядків при зворотному з'єднанні `JOIN`).
**Аналіз:** Класичне порушення 5NF — потрійний зв'язок (Ternary Relationship). Якби ми записували все в одну таблицю: `(Пацієнт, Лікар, Послуга)`. Якщо розбити її на пари `(Пацієнт, Лікар)` та `(Лікар, Послуга)`, при з'єднанні ми б отримали записи про послуги, які лікар може надавати, але не надавав *конкретному* пацієнту.
**Висновок:** Наша архітектура розбита семантично правильно: `appointment` (Лікар + Пацієнт + Час) пов'язується з `appointment_details` (Прийом + Послуга). Така конструкція не генерує хибних з'єднань і **задовольняє вимоги найвищої практичної форми — 5NF**.

## 4. SQL-скрипти (DDL) для переходу до нормалізованої схеми

Команди `ALTER TABLE` та `CREATE TABLE`, які реалізують описані вище кроки декомпозиції для приведення бази до повної 3NF/5NF.

```sql
-- ==========================================
-- 4.1. Усунення множинних значень (1NF)
-- ==========================================

-- Створення довідника спеціальностей
CREATE TABLE specialty (
    specialty_id SERIAL PRIMARY KEY,
    title VARCHAR(50) NOT NULL UNIQUE
);

-- Створення таблиці-зв'язки
CREATE TABLE doctor_specialty (
    doctor_id INT NOT NULL REFERENCES doctor(doctor_id) ON DELETE CASCADE,
    specialty_id INT NOT NULL REFERENCES specialty(specialty_id) ON DELETE CASCADE,
    PRIMARY KEY (doctor_id, specialty_id)
);

-- Видалення неатомарного поля з основної таблиці
ALTER TABLE doctor DROP COLUMN specialization;


-- ==========================================
-- 4.2. Усунення транзитивної залежності (3NF)
-- ==========================================

-- Створення довідника категорій послуг
CREATE TABLE service_category_dict (
    category_id SERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE
);

-- Підготовка таблиці послуг
ALTER TABLE service ADD COLUMN category_id INT;

-- Застосування обмежень (Foreign Key) та видалення застарілого поля/типу
ALTER TABLE service 
    ADD CONSTRAINT fk_service_category FOREIGN KEY (category_id) REFERENCES service_category_dict(category_id),
    DROP COLUMN category;

-- Очищення бази від старого ненормалізованого типу
DROP TYPE IF EXISTS service_category;
