# Лабораторна робота №6 | Робота з Prisma ORM та міграції бази даних

Цей файл є повним звітом про виконання лабораторної роботи №6. У ньому задокументовано процес налаштування середовища, розв'язання технічних конфліктів та опис проведених міграцій у схемі медичної клініки.

---

## Теоретична частина: Що таке міграції?

**Міграції** — це "система контролю версій" для вашої бази даних. Замість того, щоб вручну змінювати таблиці через SQL-запити в консолі, розробник описує бажану структуру у файлі `schema.prisma`.

* **Навіщо вони потрібні:** Міграції дозволяють автоматизувати процес оновлення бази. Це критично важливо при роботі в команді: коли один розробник додає поле в таблицю, інший може просто "запустити міграцію" і отримати ідентичну структуру без ручного налаштування.
* **З чим їх "їдять":** Prisma порівнює поточний стан бази даних із вашим кодом у файлі схеми. Якщо вони відрізняються, Prisma генерує необхідні команди для приведення фізичної бази у відповідність до вашого коду.

## Звіт про внесені зміни (Migration Notes)

Нижче наведено детальний опис модифікацій, проведених у рамках цієї лабораторної роботи для медичної інформаційної системи.

### 1. Додавання нової таблиці (medical_record)
**Опис:** Створено модель для зберігання історії хвороби пацієнтів.
**Встановлено зв'язки:** Реалізовано зв'язок "один-до-багатьох" (1:N). Один пацієнт (`patient`) може мати безліч медичних записів, що дозволяє вести хронологію лікування.

```prisma
// НОВА МОДЕЛЬ У ФАЙЛІ SCHEMA.PRISMA
model medical_record {
  record_id   Int      @id @default(autoincrement())
  description String   @db.Text
  created_at  DateTime @default(now())
  patient_id  Int
  patient     patient  @relation(fields: [patient_id], references: [patient_id])
}

// ОНОВЛЕННЯ МОДЕЛІ PATIENT ДЛЯ ЗВОРОТНЬОГО ЗВ'ЯЗКУ
model patient {
  patient_id     Int              @id @default(autoincrement())
  // ... існуючі поля
  medical_record medical_record[] // Зворотній зв'язок для 1:N
}
```

### 2. Модифікація таблиці people
Додано нове поле `address` для розширення контактних даних пацієнтів та персоналу клініки.

**Оновлена структура моделі:**
```prisma
model people {
  people_id    Int      @id @default(autoincrement())
  first_name   String   @db.VarChar(25)
  last_name    String   @db.VarChar(32)
  address      String?  @db.VarChar(255) // Нове необов'язкове поле (Optional)
  birth_date   DateTime @db.Date
  phone_number String   @unique @db.Char(12)
  email        String   @unique @db.VarChar(64)
}
```

### 3. Вирішення технічних конфліктів (Enums & Ports)
В ході роботи було виконано критичні налаштування для забезпечення сумісності Prisma з оточенням:
* **Адаптація кирилиці:** Оскільки Prisma не підтримує кириличні назви змінних у коді, було використано атрибут `@map`. Це дозволило використовувати англійські назви в JavaScript (наприклад, `MALE`), зберігаючи при цьому українські значення в самій базі даних (`"чоловік"`).
* **Конфлікт портів (Conflict Resolution):** Через конфлікт із фоновими службами Windows, стандартний порт PostgreSQL `5432` був змінений на `5433` у файлі `docker-compose.yml` та `.env`. Це дозволило Prisma успішно отримати доступ до Docker-контейнера.

##  Перевірка та верифікація результатів

Для підтвердження успішного виконання роботи використано інструмент **Prisma Studio** та тестовий скрипт `index.js`.

**Результати перевірки:**
* **Цілісність:** Всі зв'язки (Foreign Keys) між лікарями, пацієнтами та персоналом збережені та функціонують.
* **Результат запиту:** Скрипт `index.js` успішно підключився до бази та повернув список пацієнтів (Мар'ян Місюрко, Влад Гілль, Рома Волощин та ін.) у форматі JSON прямо у термінал.
* **Синхронізація:** Команда `npx prisma generate` успішно оновила клієнтську бібліотеку під нову структуру.

##  Інструкція для репозиторію GitHub

Для повної здачі проєкту в репозиторій завантажено наступні компоненти:

1.  **`prisma/`** — папка зі схемою `schema.prisma`.
2.  **`index.js`** — робочий скрипт для перевірки з'єднання.
3.  **`package.json`** — опис залежностей проєкту.
4.  **`README.md`** — цей розгорнутий звіт.
5.  **`.env.example`** — приклад налаштування підключення без паролів.
6.  **`docker-compose.yml`** — конфігурація Docker-середовища.

> **Примітка:** Папка `node_modules` ігнорується системою Git (додана в `.gitignore`), оскільки вона генерується локально командою `npm install`.
